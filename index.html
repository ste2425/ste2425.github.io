Implementing a middleware layer with Fetch()

Quite often in our apps we find the client performing more logic and deciding itself what network calls to make for the various reources it needs. This means JavaScript requires some mechanism for initiating network requests.

Originally there was XMLHttpRequest. The history of this is quite interesting, originally created Microsoft. A working draft specification was not created untill as late as 2006, but thats a story for a ifferent day.

Now there is Fetch, with a much simpler Promise based API. It will be familier to those used to jQuery or Angular.JS wrappers for XMLHttpRequest.

A usefull feature that frameworks such as Angular.JS implement is middleware, or imterceptors. This is a mechanism to inject functionality in the request pipeline. Possibly to add headers to a request, re-issues a failed request or to add logging.

Now Fetch doesn’t have this functionality out of the box but thanks to the Promise spec it is quite easy.

Concept

There will be four middleware pipelines, which will consist of an array of promises each. It will be a waterfall effect with the previous promise feeding into the next.

Wrapping fetch

In order to inject the pipelines we need to wrap fetch. There are a few ways we could do this.

We could have a piece of code that we run right at the very begining that will overrie the global reference to window.fetch (assuming you in the browser and not nodeJS or a web/service worker).

However thats messy, overriting global references and having to execute scripts in a spcific order is a step backwards.

Instead creating a class would be preferable. This means it doesn’t mutate other references. It can be used and disposed of when not needed also it can also be mocked when testing.

Multiple Enviornments

All the wrapping class needs to do is store a reference to fetch. However this can be a little more convoluted then it appears. Multiple environments expose fetch is different ways.

In the browsers its against the window object. In a web worker its against self, in Node and react native its against global.

Environment Detection

The environment could be detected based on objects that exist. With that information the fetch object could be retrieved.

There are problems with this depending on the CI envrionemtn used. Some testing frameworks expose a window object even though they are not a browser based environment.

A better solution would be to provide server and client entry points then internally access to fetch object accordingly. A browser and man entry point can be specified with the package.json if thi is going to be an NPM package.

See here for webpack PR implementing such support

Putting that all together so far

O.K. so if your still here lets summarise what weve got so far.

We have our wraping class, getting the fetch instance based on the current environment. However you will notice the constructor now takes a few new arguments. They are our four pipelines. The thing to note here is we default the error pieplines which a single promise that rejects what ever data it recieves. This is important and we will expand on that later.

Implementing the pipes

To implement the pipes we simply need a method of executing an array of promises in series. Behaving much like Promise.all() except in series.

What this does is recieve an array and reduce that down into a promise chain. The way it works the values within the array don’t have to be promises either. This util method returns a function to create the promise chain. That function rcieves a singl argument which is the initial value to to be wrapped in a promise.

Earlier it was mentioned that the class needed a _reject method. This is because of our promiseSeries implementation. you may notice the initial value is a resolved promise. This means if that is passed into a .catch method it will resolve the promise. This mans the implementation thinks we have fixed the error and so continues the chain. If no error handling middleware has been provided we need it to reject the promise chain so it will error. Defaulting the error pipelines to a instantly rejecting promise ensures this behaviour.

So using that in our fetch() method.

This solves the first pipeline. When a request comes in, first all the configuration middlewares will be executed in series. The resolution of that promise chain will be what the native fetch instance will recieve and execute the request on.

There is one caveat to note here. Promises resolve with a single argument. However fetch can recieve upto two arguments, the URL and a configuration object. In order to handle this the pipeline promise chain recieves the arguments array directly, not the individual items. Throughout the chain the pipeline must return the array.

Once complete we destructure the final arguments to execute fetch.

With that complete its quite easy to chain the remaining pipelines. With one important note, the request pipeline and error handling needs to be a nested chain verses to configuration.

If the chain is flat, the configuration error handler may re-throw the rejected promise. That means the next handler in the chain will be the response error handling meaning it will attempt to handle the error.

If the response pipelines are nested, when the configuration error handler re-throws the error the response pielines will not be executed, it will be shortcurited (much like an if statement expression).
